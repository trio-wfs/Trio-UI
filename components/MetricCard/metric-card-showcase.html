Metric Card

Metric Cards display numeric KPIs and data summaries in a compact 150px-tall card. They support 1–4 metric values, four footer styles, and per-metric semantic color overrides. Cards are always used in horizontal groups of 2 or more, filling available width with 16px gaps between them.

category Custom Component verified Production Ready

info Overview

bar_chart Purpose
Display compact numeric KPIs at a glance. Each card shows a label, 1–4 metric values with sub-labels, and an optional footer (progress bar, icons, or labels). Height is always exactly 150px regardless of variant.
palette Color System
Uses the info tone scale (900 → 700 → 500 → 300) as the primary data viz color. Semantic colors (error, warning, success) can override any metric starting at 500, ascending to 700, then 900 for additional urgency.
grid_view Layout Rules
Always used in groups of 2+ cards in a horizontal flex row with 16px gaps. Cards have a minimum width of 170px and expand to fill available space equally. Labels truncate with ellipsis at minimum width.

rule Usage Guidelines

check_circle When to Use
Correct
  • Dashboard KPI summaries (counts, totals, statuses)
  • 2+ related metrics that benefit from side-by-side comparison
  • Proportional data that a progress bar meaningfully represents
  • Operational metrics with urgency states (error/warning/success)
  • Grouped comparison: 1 primary metric + 2 supporting sub-metrics
  • Lists of 4 related values that don't need chart visualization
cancel When NOT to Use
Incorrect
Single standalone card without context
  • Single standalone card — always pair with at least one other card
  • More than 4 metric values — use a data table instead
  • Trend over time — use a chart component
  • Detailed breakdowns — use a table or expandable list
  • Non-numeric content — use a summary card pattern instead
  • Mixing info and semantic colors randomly — use per-metric overrides intentionally

widgets Metric Count Variants

The number of metrics in the array drives the layout. 1–3 values display as equal-width columns. 4 values switch to a vertical list. Value font size scales down to fit (34px → 24px → 16px) as more columns are added.

1 Metric — Single Column (fs=34, info/900, bar footer)
2 Metrics — Two Equal Columns (fs=34, info/900 + info/700)
3 Metrics — Three Equal Columns (fs=24, info/900 + 700 + 500)
4 Metrics — Vertical List (fs=16, list layout, no footer)

view_headline Footer Variants

The footer area below the metric columns is swappable. The bar is the default. Icons and labels can replace the bar. Any footer can be omitted entirely.

footer="bar" — Proportional segmented progress bar (default)
footer="icons" — Per-metric Material icon indicator
footer="labels" — Secondary caption labels (Figma: style=custom)
footer="none" — No footer, metrics fill available height

view_sidebar Grouped Layout

The grouped layout places one large primary metric on the left (34px) and stacks 2 smaller caption/value pairs on the right (16px). This is the Figma "groupe/bar" variant. Supports bar or no footer.

layout="grouped" with bar footer
Total + breakdown — excludePrimaryFromBar

When the primary metric is the sum and the remaining metrics are its breakdown, set excludePrimaryFromBar so the progress bar segments only the breakdown. Without this flag, BarFooter sums every metric and the bar double-counts the total against its parts (e.g. [14, 7, 7] renders as 50/25/25 instead of 50/50).

Tappable metrics — onClick per item

Individual metrics become focusable <button> elements with a pointer cursor when you pass onClick on the metric item. A single card can drive multiple filter targets — each value fires its own handler. Non-clickable metrics stay as plain <div> with no cursor change. The "total" stays non-clickable on purpose; clicking the sum isn't a filter, it's the unfiltered state.

No selected / hover / pressed visuals on purpose — the affordance is the cursor and the browser-default focus ring. Keyboard users get Tab + Enter / Space for free. Live demo isn't shown here because onClick is a function reference that can't be expressed in static JSON props; see the Privilege Activity prototype for the wired-up version.

toggle_on States

Any metric can be assigned a semantic color override (error, warning, success) for urgency signaling. Semantic colors start at 500 for the first occurrence and ascend to 700, then 900. Info metrics retain their positional tone assignment. You can mix info and semantic colors within the same card.

All metrics → error (500 → 700 → 900)
Mixed — info (1st metric) + error override (2nd metric)

label Label Icon Variants

The label row supports an optional left icon (Material Icons only) or an optional help_outline (?) icon on the right — never both. The left icon is a decorative scanning aid; the (?) is functional and anchors a tooltip. They compete for attention in the 24px label band, so pick one based on the card's job. If both props are passed, the component suppresses the left icon and the (?) wins.

recommend Best Practices

check_circle Do
Correct
Use groups of 2+ cards with consistent footer styles
  • Primary data viz uses the info tone scale — positions assigned info/900 → 700 → 500 → 300. This is the default; don't paint metrics blue or in arbitrary palette picks.
  • Semantic colors signal urgency, not decoration — use error / warning / success only when the metric value itself communicates state (lapsing credentials, expired licenses, fill-rate misses).
  • MetricCard is a standalone component — do NOT wrap it in a ContentContainer. It owns its own surface, border, and padding. Same rule applies to Alert and Stepper.
  • In responsive layouts, MetricCard rows snap from 4-up to 2-up to 1-up at md / sm breakpoints — never gradually reflow.
  • Use excludePrimaryFromBar when the primary metric is the total — the bar should segment only the breakdown (Active + In Progress), not double-count the sum against its parts.
  • Pass onClick on individual metrics[i] to make a value a filter trigger — each metric becomes its own focusable button. Lets one card drive multiple filter targets without a card-level handler.
  • Always use cards in groups of 2 or more — a single card lacks the comparative context the format provides
  • Use consistent footer style within a card group
  • Keep labels short — they truncate at minimum width (170px)
  • Use bar footer for proportional data, icons for status indicators
cancel Don't
Incorrect
Mixing footer styles within the same group
  • Don't paint MetricCard values with primary blue (#2196F3) — primary is reserved for Save/Update actions and links/clickable text. Use the info tone scale (default) or semantic colors as called for.
  • Don't apply semantic colors decoratively — coloring a static count success green because it "looks healthy" breaks the rule. Semantic = state, not aesthetic.
  • Don't wrap MetricCard in ContentContainer — it's a standalone component, not section content.
  • Don't show the left labelIcon and the (?) help icon together — pick one based on whether the card needs a scanning aid or a tooltip
  • Don't make the "total" primary metric clickable in a tappable grouped card — clicking the sum isn't a filter, it's the unfiltered state. Only the breakdown values should fire onClick.
  • Don't sum the primary into the bar footer when it's a total — without excludePrimaryFromBar, a [14, 7, 7] card renders as 50/25/25 instead of the intended 50/50 split.
  • Don't use a single standalone card without context — cards live in groups
  • Don't mix footer styles within the same card group
  • Don't use more than 4 metrics — switch to a table
  • Don't mix info and semantic colors randomly
  • Don't use for trend data — use Chart component instead

Prop Type Default Description

showHelpIcon boolean false Show help_outline icon at right of label row. Suppresses labelIcon when true.

onClick (e) => void — Card-level click handler. When set, the whole card becomes clickable (cursor: pointer). For per-value filter triggers, use metrics[i].onClick instead.

onClick (e) => void Makes this individual metric act as a focusable <button> with a pointer cursor. Other metrics on the same card stay non-clickable unless they also set onClick. No selected / hover / pressed visuals — affordance is the cursor + browser focus ring.
