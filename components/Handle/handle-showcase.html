Handle

Drag-handle affordance for reorderable lists and rows. Signals "grab me to move" with a six-dot icon that expands on hover and grows again during the drag.

category Interaction Affordance verified Production Ready

info Overview

psychology Purpose
A small grab target that tells the user "this row is draggable." Used at the leading edge of a reorderable row so the rest of the row stays clickable for normal interactions.
priority_high Visual Hierarchy
Low — Handle is a chrome affordance, not a control. It sits in the row gutter at #424242 and only lights up on hover so it never competes with row content.
accessibility Accessibility
Renders as role="button" with aria-label="Drag to reorder". Pair with a keyboard reordering pattern (e.g., select + ArrowUp/ArrowDown) since drag-and-drop alone is not keyboard-accessible.

rule Usage Guidelines

check_circle When to Use
Correct
Hospital Network Step 1
Facility Tier Step 2
  • Reorderable lists where order matters (workflow steps, custom column order, priority lists)
  • Settings rows users can rearrange (rule precedence, distribution waterfall)
  • Wizard/builder steps where users reshape the sequence
  • Saved-view ordering in a sidebar or settings panel
  • Any list where the row already has primary actions and drag must not steal clicks
cancel When NOT to Use
Incorrect
Shift #4821 Submitted
  • Sortable data grids — column sorting and row order live in AG Grid, not Handle
  • Single-row selection or expand/collapse — use a ButtonIcon or row click instead
  • Static lists where order isn't user-editable — don't suggest interaction that isn't there
  • Mobile-first surfaces — touch drag is unreliable; use up/down arrow buttons instead
  • Lists with fewer than 3 items — reordering isn't worth the affordance cost

toggle_on States

Three visual states sized to communicate intent. The default sits quietly in the row gutter; hover lights up the icon to confirm grab-ability; drag grows the target for confident motion. Use the state prop only for showcase or Storybook — in production, hover and drag are CSS-driven.

Default — 24×24
Hover — 24×24 + tint
Drag — 32×32 + shadow
Disabled

recommend Best Practices

check_circle Do
Correct
Worker rating priority
  • Place Handle at the leading edge of the row, before any content
  • Provide a keyboard reorder fallback (select + ArrowUp/ArrowDown) — drag alone isn't accessible
  • Use 8px row gutter spacing so the hover/drag growth doesn't shift surrounding content
  • Persist the new order immediately on drop — reordering shouldn't require a Save button
  • Animate the row drop with a brief fade or settle so the user sees where it landed
cancel Don't
Incorrect
Step 1
  • Don't put Handle at the trailing edge of the row — the leading edge is the established pattern
  • Don't show Handle on rows that aren't actually reorderable — it's a promise
  • Don't override the cursor — grab / grabbing is part of the affordance
  • Don't restyle the icon — the six-dot grid is the universal drag-handle signal
  • Don't combine Handle with row-level checkboxes that toggle on row click — clicks compete
