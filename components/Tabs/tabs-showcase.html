Tabs

Horizontal navigation that switches between related content panels on the same page. Use tabs when users need to switch between a small number of related views without leaving the page.

category Navigation Component verified Production Ready

info Overview

psychology Purpose
Horizontal navigation between related views within a section. Used when a page has multiple distinct sub-views that share context (worker profile tabs, settings sections).
priority_high Visual Hierarchy
Medium — Tabs orient the user within a section. Active tab matches the surrounding content's hierarchy.
accessibility Accessibility
Native role="tablist" with arrow-key navigation between tabs. aria-selected="true" on the active tab. Tab key moves into and out of the tablist.

rule Usage Guidelines

check_circle When to Use
Correct
  • Pattern A — detail page section tabs at the content wrapper level (Worker, Order, Facility detail). Use surface="secondary" — the active tab blends with the #FAFAFA wrapper. Switching a tab swaps the entire layout below, including rails.
  • Pattern B — in-container tabs inside a paper ContentContainer, switching between views of the same data set (e.g. Active / Completed / Cancelled inside a card). Use surface="paper".
  • Subject identity (title, chips, eyebrow) lives in the PageHeaderToolbar — not repeated on every tab.
  • Keep tab count between 2 and 8 — cognitive load climbs past that
  • Use scroll variants when tab count is variable — never wrap to a second line or truncate labels.
cancel When NOT to Use
Incorrect
  • Pattern C — grid view switching is NOT Tabs. The All | Active | Pending | … switcher on a standalone grid page is the Breadcrumb component in state="Links", living in the PHT's breadcrumbContent slot. Same height, same #FAFAFA chrome, pipe dividers.
  • Don't use Tabs as header tabs on grid pages — standalone grid pages use the PHT full variant with Breadcrumb-Links in the breadcrumb slot.
  • Don't use Tabs for sequential workflows — use a Stepper instead
  • Don't use Tabs when all content must be visible at once — tabs are for parallel, mutually exclusive views
  • Don't exceed 8 tabs — use secondary navigation or grouping
  • Don't put tabs inside modals unless absolutely necessary — break complex forms into Stepper steps instead
  • Don't use long, descriptive labels — keep to 1-2 words in title case
  • Don't wrap tabs to a second line, truncate, or hide overflow in a dropdown — use the scroll-arrow variants.

layers Surface

The active tab fills with the surrounding page background so the selected tab visually connects to the content below. Use surface="secondary" when the tab strip sits over a secondary-tinted background (e.g. a settings page or nested panel).

recommend Best Practices

check_circle Do
Correct
  • Tabs sit flush on their surface — no padding around the tabs themselves. Content below gets 16px padding.
  • The selected tab's primary-blue indicator is at the TOP, not the bottom — paired with rounded top corners and 1px #E0E0E0 left/right strokes. The selected tab reads as a card-tab attached to the content below.
  • Tab icons are MUI Outlined at 16px — never another icon library, never another size.
  • Keep tab labels short (1-2 words) in title case — users scan, they don't read
  • Use icons only when they add clarity to the label — decorative icons add noise without aiding recognition
  • Place tabs at the top of the content area they control — never floating mid-page
cancel Don't
Incorrect
  • Don't put the indicator on the bottom edge of the tab — the canonical position is the top edge, matching the card-tab attachment metaphor.
  • Don't pad around the tab strip itself — tabs sit flush on their surface; padding belongs around the content panel below.
  • Don't use Tabs for sequential workflows — use Stepper
  • Don't exceed 8 tabs — cognitive load increases significantly
  • Don't put tabs inside modals unless absolutely necessary
  • Don't use for content that must all be seen together
  • Don't use long, descriptive tab labels — keep them scannable
