Button Group

Groups related button actions together to show multiple options in a single, connected unit. Use to present related actions that should be visually grouped.

category Atomic Component verified Production Ready

info Overview

psychology Purpose
Joins 2–4 related actions into a single visual cluster with shared border treatment. Used when actions share an axis — segmented save/cancel, formatting controls, view toggles.
priority_high Visual Hierarchy
Inherits the hierarchy of its constituent buttons. Visual grouping does not change individual button weight.
accessibility Accessibility
Each button inside the group retains independent focus and its own accessible label. Tab moves between them; arrow keys do not — they remain independent buttons, not a roving tablist.

rule Usage Guidelines

check_circle When to Use
Correct
  • Group related actions that modify the same object
  • Use 2-4 buttons for optimal usability
  • Use clear, action-oriented labels
  • Use primary color only for confirming/save actions
  • Use secondary for editing/prompt actions
cancel When NOT to Use
Incorrect
  • Don't group unrelated actions together
  • Don't exceed 6 buttons (Figma limit: button-8)
  • Don't use for exclusive selection (use Toggle Button)
  • Don't mix primary and secondary colors in one group
  • Don't use for navigation (use Tabs instead)

widgets Variants

Two surface pairs (contained + secondary for neutral grouped actions, outline + primary for action-emphasis groups) at two sizes (small 32px, medium 38px), in horizontal or vertical orientation. variant determines color — the other combinations don't exist in the design system and are TypeScript errors at the call site.

Surface pairs — variant determines color
Small (32px) — toolbars and dense interfaces
Medium (38px) — standard forms and modals
Vertical — for narrow containers
With icons — each entry can carry startIcon / endIcon

toggle_on States

Button group states mirror individual button states. Hover any button in the group to see the interactive state.

recommend Best Practices

check_circle Do
Correct
  • ButtonGroup is what goes in the PHT's buttonGroupContent slot — connected secondary actions like the icon trio (Filter / Columns / More) or labeled clusters (Export / Print). Multiple ButtonGroups can sit in the actions area when actions cluster naturally.
  • The page's primary action is NOT a ButtonGroup — it's a single Button in the PHT's singleButtonContent slot (e.g. "+ New Worker").
  • Use small size for grid filter rows and dense toolbars (32px = tokens.controls.height.small); medium (38px) for primary action surfaces.
  • Use the verb-noun pattern for ButtonGroup button labels too — Export CSV, Print Page, Filter Active. Never OK, Submit, Save.
  • Group 2-4 related actions together — the segmented chrome implies "these belong"
  • Use consistent color within a single group
  • Use primary blue (#2196F3) for Save/Update actions only — never for navigation or general grouping
cancel Don't
Incorrect
  • Don't put the primary action inside a ButtonGroup of secondary actions — the primary lives in the PHT's singleButtonContent as its own Button.
  • Don't pour loose Button / IconButton instances into the PHT actions area — wrap them in ButtonGroup so they share the segmented chrome.
  • Don't hardcode button heights inside the group — pull from tokens.controls.height so the size matches every other 32/38px control in the system.
  • Don't mix primary and secondary colors in one group — the segmented chrome reads as "these all share priority"
  • Don't exceed 6 buttons in a single group
  • Don't use ButtonGroup for exclusive toggle selection — use ToggleButton
  • Don't use ButtonGroup for page navigation — use Tabs (Pattern A or B) or Breadcrumb-Links (Pattern C)
  • Don't group unrelated actions together
