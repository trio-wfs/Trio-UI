NumberField

Stepper input for numeric values with bounded min/max ranges. Use for quantity selection, rate adjustments, and any small-range integer input where users may prefer clicking over typing.

category Form Control verified Production Ready

info Overview

psychology Purpose
A composite control — minus button, centered input, plus button — for entering a small numeric value. Differs from a plain TextField by giving users an obvious tap-to-step affordance for low-cardinality numeric entry.
priority_high Visual Hierarchy
Low — NumberField sits within forms as a peer of TextField, Select, and Switch. Use optional label + helperText to give it the same visual weight as adjacent inputs.
accessibility Accessibility
Supports keyboard entry, ArrowUp/ArrowDown for stepping, and Enter to commit. Buttons expose aria-label and disable at min/max boundaries. Pair with a visible label for screen-reader context.

rule Usage Guidelines

check_circle When to Use
Correct
  • Bounded integer entry where the typical value is 1–20 (workers needed, beds, shifts)
  • Quantity selection in modals (number of submissions, copies of a template)
  • Rate or threshold tuning with small steps (overtime multiplier, alert window in hours)
  • Any numeric value where users may want to nudge up/down rather than type
  • Inputs constrained to a clear min/max range — the stepper makes boundaries obvious
cancel When NOT to Use
Incorrect
  • Large numeric ranges (currency, dates, percentages with many decimal places) — use TextField with numeric validation
  • Currency input — use a TextField with adornments and formatting
  • Search-like numeric entry where typing is faster than stepping — use TextField
  • Numeric values selected from a known list — use Select or Autocomplete
  • Continuous-range tuning where exact value matters less than position — use Slider

widgets Variants

NumberField has one shape — a stepper with a centered numeric input. Optional label and helperText slots adapt it to match the form context. Use the bounded version (with min/max) whenever the valid range is known — the boundary buttons disable automatically.

recommend Best Practices

check_circle Do
Correct
Specific label + bounded range + helper text describing the constraint
  • Set min and max whenever the valid range is known — the boundary buttons disable automatically
  • Use specific labels tied to the domain (“Workers needed” not “Number”)
  • Add helper text that names the constraint or unit (“hours”, “0 to 10”)
  • Pair with an aria-label when the visible label is omitted for layout reasons
  • Use step for non-integer increments (e.g., step=0.5 for half-hour increments)
  • Match Button size in the same form — medium with medium buttons, small with small
cancel Don't
Incorrect
Vague label + unbounded + large value — this should be a TextField
  • Don't use for currency, salary, or any value users type rather than step — use TextField
  • Don't leave bounded inputs unbounded — users may step into invalid values before error appears on submit
  • Don't use unlabeled instances in primary forms — reserve label-less use for grid cells and toolbar controls
  • Don't omit helper text when the constraint isn't obvious from the label alone
  • Don't override the centered input alignment — the stepper symmetry is intentional
