Modal

Two fixed sizes for dialogs, confirmations, and forms. Overlay blocks background interaction with 50% black scrim.

category Overlay Component verified Production Ready

info Overview

psychology Purpose
Full-attention overlay that blocks the page until dismissed. Used for confirmations of consequential actions, focused forms, and destructive operations that require explicit acknowledgment.
priority_high Visual Hierarchy
Highest in the app when open. Modal takes precedence over everything beneath it and must be addressed before the user can continue.
accessibility Accessibility
Focus is trapped inside the modal. Escape closes (unless explicitly forbidden). aria-modal="true" and aria-labelledby on the dialog. Focus returns to the trigger on close.

widgets Variants

Two fixed widths based on content complexity. Small for simple interactions, Large for multi-field forms and data entry.

Small Modal — 550px

Use for confirmations, alerts, simple forms, and single-step actions.

Large Modal — 900px

Use for multi-field forms, data entry, complex workflows, and feature-level dialogs.

Destructive — red confirm

Use variant="destructive" for irreversible actions: delete, revoke, terminate, remove. The confirm button switches to color="error" to make the destructive intent obvious.

biotech Anatomy & Spacing

Both sizes use 40px left/right padding across all zones, with 16px top/bottom on header and footer, and 24px top/bottom on the body. A 16px gutter separates content columns. The 12-column grid inside a modal scales to the modal's content width.

height Height & Scrolling

Modals do not have a fixed height. The height is driven by the content inside — short forms produce short modals, long forms produce tall ones. The modal will expand until it reaches the limit of the browser window, then stop and scroll internally.

The viewport is the browser window
The modal has no awareness of your screen resolution or monitor size. Its available height is exactly the height of the open browser window at the moment the modal appears. If the user has a short window, the modal has less room. If they have a tall window, the modal has more.
Max-height caps at 90% of the window
The modal will never grow taller than 90vh — 90% of the browser window height. This leaves a visible gap at the top and bottom so the user always knows they're in a modal layer, not a full page. The remaining 10% is breathing room, not wasted space.
Header and footer are always visible
No matter how long the form is, the blue header (title + close button) and the toolbar footer (Cancel + Save) are anchored in place. They do not scroll with the content. The user can always dismiss or save without scrolling to the top or bottom.
Only the body scrolls
When a form is taller than the available space, only the middle content area scrolls. Think of it like a sandwich — the top slice (header) and bottom slice (footer) stay put, and only the filling moves. Engineers implement this with overflow-y: auto on the body region only.

recommend Best Practices

check_circle Do
  • Use the verb-noun label pattern — Create [Noun] for new entities, Update [Noun] for changes to existing ones (e.g. "Create User", "Update Bill Rate"). See PAGE_ARCHITECTURE.md §16 for the full table.
  • Use specific verb labels for non-CRUD actions — "Reject submission" beats "OK"; "Delete account" beats "Confirm"
  • Use variant="destructive" for irreversible actions so the confirm button switches to red
  • Auto-focus the Cancel button or the first form field — never the destructive action
  • Provide all four dismissal paths: X button, Cancel button, Escape key, and scrim click
  • Keep the title concise and action-oriented — "Confirm assignment" beats "Assignment"
  • Pin the header and footer; only the body region scrolls (90vh max height)
  • Restore focus to the trigger element when the modal closes — never to page top
cancel Don't
  • Don't disable Cancel during async submission — let users back out, even mid-request
  • Don't auto-focus the destructive button — Cancel must be the keyboard default
  • Don't use generic "OK / Cancel" labels — they hide the actual consequence
  • Don't allow the entire modal to scroll — header and footer stay pinned to anchor actions
  • Don't nest a modal inside a modal — collapse the deeper interaction or use a Drawer
  • Don't add extra animation flourishes — the 300ms fade-slide already exists, more is noise
  • Don't block dismissal for routine flows — reserve no-escape only for genuinely irrecoverable in-flight work
