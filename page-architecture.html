TRIO WFS Page Architecture

Rules for how pages are assembled from components. Read alongside design-tokens/tokens.ts and the component library. Token values are owned by tokens.ts — this file references token roles and usage.

01Page anatomy — layer order

Every TRIO page is built from three background layers, top to bottom. Each layer has a single canonical role — never mix them up.

background.default · #F5F5F5 · page canvas
Page title
⋯ + New
background.secondary · #FAFAFA · content wrapper
background.paper#FFFFFF
background.paper#FFFFFF
paper inside paper

Rules

  • Page canvas #F5F5F5 — only for the page canvas. 16px padding on all four sides — the PHT and content wrapper sit within this padded area, never running to the browser edge.
  • Content wrapper #FAFAFA — the single wrapper below the PHT. Same border treatment as paper: 1px solid #E0E0E0 + 4px radius. Tabs sit flush at its top.
  • Paper #FFFFFF — all cards, panels, containers. Paper can nest inside paper. No exceptions.
  • Borders on every container surface: 1px solid #E0E0E0. No elevation.

02Elevation & shadows

Drop shadows are only used on floating / overlay elements:

  • Modals
  • Menus
  • Popovers
  • Breadcrumb dropdowns

Never on cards, panels, containers, or any paper surface that's part of the page layout. Token: tokens.shadows.sm.

03Navigation header

The NavigationHeader sits fixed at the top of every page. Always present, always the same — no variants.

  • 16px padding-bottom separating it from page content
  • Exception: Digital Workers and future AI layouts omit the NavigationHeader entirely

Floating vs attached

The toolbar either floats (16px gap to content) or attaches (no gap, shared border with content below). The choice depends on whether the toolbar controls the content directly.

Floating · 16px gap

Standalone bar with title and actions. Content below is independent — cards, panels, forms. Default behavior.

Attached · no gap

The breadcrumb row is swapped for sub-section tabs that control the grid below. The toolbar and grid share one continuous border — they read as one visual unit.

Action slot composition

The right side has three explicit slots — use them for what they're named for. Don't pour loose buttons into the actions area.

SlotComponentUse for
inputTextFieldTextField / SearchBarSearch or filter input scoped to the page
buttonGroupButtonGroup (one or more)Connected secondary actions — icon trios (filter / columns / more), or labeled clusters (Export / Print)
singleButtonButtonThe single primary action for the page — "+ New Worker", "Save Changes"
Don't
Don't drop loose Buttons next to each other in the actions area — group them via ButtonGroup so they share one outer border. Don't put primary actions inside a ButtonGroup of secondary actions — the primary is its own Button in singleButton.

Key rules

  • Header tabs are only for grid switching — tabbing between AG Grid views on list pages
  • Detail pages use the Tabs component at the content wrapper level for section navigation — never header tabs. See §11.
  • Never invent a new header layout — always use a variant of PageHeaderToolbar
  • If a page has no actions, remove the button group entirely rather than disabling buttons

05Color usage

Background hierarchy

  • Never use #F5F5F5 for a card or container — that's the page canvas only
  • Never use #FAFAFA for a card — that's the content wrapper only

Primary blue #2196F3

  • Save and Update actions only
  • Not for navigation, not for status indicators, not for decorative use

Data viz teal #37636b

  • Metric card bars, stepper indicators, charts
  • Complements the primary blue palette without competing with it
  • Not for buttons, not for status chips
Stepper drift
Active step circle, completed checkmark, and connectors between completed steps all use teal. The current Stepper.tsx uses primary blue — that's a known drift to be aligned to this rule, not the rule to follow.

Semantic colors — status & feedback only

ColorTokenValueUse for

What never gets a color

  • Page backgrounds (always default / secondary / paper)
  • Borders (always #E0E0E0)
  • Dividers (always #E0E0E0)

06Layout

Standard pages

  • 16px padding inside the background.secondary wrapper
  • 12-column grid, 16px gutters
  • Desktop-first — 1440px viewport target. Layout primitives stack below 1200px (lg) per §7. This is a graceful floor, not mobile support.

Full-width pages (Digital Workers, future AI)

  • No outer 16px padding — content stretches to the browser edge
  • Header still present but frameless
  • Same paper / border rules apply inside

07Responsive floor — below lg (1200px)

TRIO is desktop-first. The intended viewport is 1440px. Below ~1200px the experience is degraded, not designed — the goal is graceful stacking, so a user on a narrower window sees something readable, not broken.

Fluid width, snap arrangement
Containers grow and shrink with the browser within a breakpoint range. A 4-up MetricCard row at 1440 is still a 4-up row at 1280 — each card just narrower. The arrangement (4-up → 2-up → stacked) only changes when the browser crosses a defined breakpoint, where it snaps.

The breakpoints

  • Default floor: lg (1200px). Most primitives stack below this.
  • Some primitives stack later at md (900px) — see table.
  • Use only MUI's standard breakpoints (xs:0, sm:600, md:900, lg:1200, xl:1536). Never invent custom pixel values.

What stacks, and when

PrimitiveStacks atBehavior
PageHeaderToolbarlg (1200)Title row and action row stack; chips/eyebrow stay inline
Filter barslg (1200)Inputs/chips wrap, then stack
MetricCard rowsmd / sm4-up at lg → 2-up at md → 1-up at sm
Two-column form sectionsmd (900)Collapse to single column
Detail page railsmd (900)Left/center/right collapse to a single stacked column

What does NOT stack

  • AG Grid tables — horizontal scroll is the fallback. If unusable below lg, show "best viewed on desktop"
  • Modals — already width-constrained (500/900px), stay centered
  • Page shell — the 12-column wrapper does not change

Navigation — needs definition

Both NavigationHeader and NavigationVertical need responsive behavior defined together (paired treatments — collapsed rail + slim header, drawer pattern). Until then, both retain full desktop layout at all widths.

08Typography

  • Font: Roboto only — no Inter, no system-ui
  • Max headline: H5 (24px) for page headers
  • Page titles: H5 or H6 depending on header mode
  • Body copy: body1 (16px) for primary, body2 (14px) for dense/secondary
  • No uppercase text in UI — except overline labels where Figma calls for it
  • No font weights above 500 (Medium) — 700 is not used in TRIO UI

How to use typography in code

  • Tokens: tokens.typography.h4, tokens.typography.caption, etc. Full catalog at design-tokens-typography.html.
  • In components: use MUI's <Typography variant="h4">. The Trio theme wires every variant to the matching token.
  • There is no Trio Typography wrapper — the theme does the work. Importing Typography from @trio-wfs/ui will not work because it isn't exported. This is intentional, not a gap.
  • Never hand-style text elements (<h1 style={{fontSize: 34}}>) — bypasses the theme and reinvents tokens.

09Component defaults

  • Border radius: 4px standard, 999px pill (chips, badges, step circles)
  • Spacing: 4 / 8 / 12 / 16 / 24 / 32 / 40 px only — no arbitrary values
  • Modals: 900px large, 500px small
  • Drawer: 400px standard width (legacy — panels on the 12-column grid are the future, see §16)
  • Icons: @mui/icons-material Outlined only — no other libraries
  • Icon size: 16x16 in buttons and tabs — always. No exceptions.
  • Data tables: AG Grid — always. Never MUI X DataGrid, never hand-rolled <table>. See §10.
  • PHT: title-to-eyebrow gap is 4px (spacing.xs)
  • Footer: sticky, 16px padding, #E0E0E0 top border
  • NavigationVertical: never has a background color — always transparent

Control heights

Every form-row control renders at standard heights. Canonical source: tokens.controls.height. No hardcoded 32 / 38 anywhere.

SizeHeightToken
small32pxtokens.controls.height.small
medium38pxtokens.controls.height.medium

Applies to: Button, Select, TextField, Autocomplete, NumberField, DatePicker, SearchBar, ButtonIcon (contained), ToggleButton (medium).

Exception — compact icon-only (24px): ButtonIcon ghost small and ToggleButton small slots use tokens.controls.ghostHeight (24px). These live in dense toolbars, grid filter rows, and breadcrumb actions where a 32px control would dominate. Figma-sanctioned exception, not a drift.

10Grid page patterns

Always use AG Grid for data tables in React. Hand-rolled <table> or MUI X DataGrid are not allowed. Static HTML prototypes mirror AG Grid's structure so engineers can drop in AgGridReact.

Structure & borders

  • Header's bottom border IS the grid's top border. The whole assembly shares one outer 1px solid #E0E0E0.
  • Vertical column dividers run through every row: border-right: 1px solid #E0E0E0 on every header th, filter cell, and data td. The last cell in each row drops its right border so the outer frame stays clean.
  • Reference: Figma node 5650:4810

Grid view switching — Breadcrumb-Links, not Tabs

  • Same 40px height, same #FAFAFA background, same caption (12px) text as a normal breadcrumb
  • Pipe | dividers (in border.default color) instead of slashes — the only visual difference from a regular breadcrumb
  • Non-selected: primary-blue link text. Hover underlines.
  • Selected: text.primary with a 2px primary-blue bottom border. Optional inline count; selected count flips to primary blue.

Column headers — paper bg, 40px tall

  • Background: #FFFFFF — not #F5F5F5
  • Height: 40px, matching the breadcrumb-Links strip above — reads as one continuous header band
  • Text: body2 medium — 14px / 500 / 21 line-height, text.primary
  • Sort icons: 16px Material Outlined. unfold_more for unsorted in text.disabled; arrow_upward / arrow_downward for active in #424242 — never primary blue.
  • position: sticky; top: 0 so the header stays under vertical scroll.

Horizontal scroll behavior

  • The .ag-grid container is overflow: auto (horizontal + vertical)
  • Table is width: max-content; min-width: 100% — grows with content, never narrower than the container
  • Each cell is white-space: nowrap — content pushes into horizontal scroll instead of wrapping
  • Per-column widths pinned via <colgroup> (or colDef.width in AG Grid)
  • PHT, breadcrumb-Links strip, and outer border stay fixed; only the grid body scrolls

Grid inside a tab

When a grid appears inside a tabbed section, the tab content area has 16px padding. The grid is wrapped in a ContentContainer. Same column-divider, paper-header, and horizontal-scroll rules apply inside.

Never use header tabs (Tabs component) for detail page section nav — that's pattern A. Never use Tabs for grid view switching — that's pattern C, which is Breadcrumb-Links.

Common rules (all three)

  • Tabs sit flush on their surface — no padding around the tabs themselves
  • Content below tabs gets 16px padding
  • Tabs with left icons are standard — use MUI Outlined icons at 16px
  • Selected tab: bottom border matches surface color (hides the HR line) and gets a 2px primary-blue indicator at the top, not the bottom
  • Selected tab has rounded top corners (4px) plus 1px #E0E0E0 left + right strokes — reads as a card tab attached to the content
  • Overflow: use the built-in scroll-arrow variant. Never wrap tabs to a second line, never truncate labels, never hide tabs in a dropdown.

Visual reference — 24px field gap

  • Form fields stack vertically with 24px (spacing.lg) gap — mandatory, not 16px
  • Two-column form grids: grid-template-columns: 1fr 1fr; gap: 24px 16px
  • Full-width fields (textareas, notes) span both columns
  • Checkboxes and radio groups: 8px (spacing.sm) between options within the group

  • Section titles use Subtitle2 (14px, Medium 500)
  • Most forms live in modals, not on their own page. If too complex for 900px, break into steps — don't make the modal bigger.

Links / clickable textPrimary blue #2196F3Email addresses, IDs that navigate, any text that triggers an action

Long text truncation

Truncate and show a "Show More" link styled in primary blue. The number of visible lines depends on context — no fixed line count.

Loading states

  • Spinners only. No skeleton loading screens.
  • If loading takes > 2 seconds, pair the spinner with a text label ("Loading credentials…")

Empty states

  • AG Grid: "No rows to display" centered. No icons, no illustrations, no action buttons.
  • Cards / pages: Explain why it's empty and suggest an action. No sad illustrations.

15Edit, delete & alert patterns

Edit & delete affordances

  • Pencil icon — standard edit trigger. Used on grid rows and inline edit actions.
  • Trashcan icon — standard remove/delete trigger.
  • Grid row editing — always opens a modal. Do not inline-edit within the grid.
  • Detail page editing — typically via modal. Default to modal unless Jesse says otherwise.

Callout / alert containers

When content needs semantic emphasis (critical requirements, compliance warnings, important instructions), wrap it in a container with a full border using the info color info.main. Not a left-accent-only treatment — full border around the container.

Use sparingly — for content the user must not miss, not general highlighting.

Alert color cohesion

When using the Alert component, every colored element inside the alert uses the same semantic color — the one matching severity. Border, icon, title, description, and action buttons all share that one color.

SeveritySingle color used everywhere
Info#54AFCA
Success#388E3C
Warning#E17109
Error#DB4537
Don't
Don't mix the semantic color (border + icon) with text.primary / text.secondary for the title and description. The alert reads as a single semantic block. (Exception: the contained filled variant where the background carries the color and text is contrastText/white — still one coordinated choice, just inverted.)

16Content surfaces — modal, panel, inline

Three ways to show secondary content. Choice depends on whether the user is completing an action, referencing information, or optionally digging deeper.

Current standard
Modal
Used for: completing actions (create / edit / save), reviewing snapshots, destructive confirmations.

Interrupts the workflow — user must complete or dismiss. Intentional for actions that change data.

Large: 900px — complex forms, multi-step, data-heavy.
Small: 500px — confirmations, simple single-action dialogs.

If too complex for 900px, break into steps — don't make the modal bigger.
Newer direction
Panel
Side panels that let the user reference detail without losing context.

Layout-driven, not a fixed-width component — use the 12-column grid (e.g. panel = 4 cols, main = 8 cols).

For referencing and reviewing, not for completing multi-field forms.

Digital Workers is the primary testing ground. Specific defaults TBD.
Optional dig-in
Inline expansion
For secondary content the user might want to dig into but doesn't need by default.

Content expands in-place — no overlay, no interruption.

Used when expanded content is subordinate to the primary content around it.

Collapse/expand is the user's choice — don't auto-expand.
Future direction
Long-term direction is moving away from modals toward more responsive, contextual experiences (panels, inline) rather than interrupting overlays. Digital Workers demonstrates this. Until proven and adopted, modals remain the default for all existing TRIO projects.
