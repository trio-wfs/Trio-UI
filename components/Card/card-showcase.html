Card

Composed surface built on Paper. Adds optional header / body / footer slots with consistent section spacing.

category Composition Primitive verified Production Ready

info Overview

psychology Purpose
Predictable section structure on a Paper surface. Header + body + footer slots with consistent 16px outer spacing.
priority_high Visual Hierarchy
Medium — Card is a content container. It frames content without competing with it. Never the focal point.
accessibility Accessibility
Pass role and aria-label when Card has semantic meaning (e.g. role="article" for posts). Otherwise plain div semantics are correct.

rule Usage Guidelines

check_circle When to Use
  • Content with discrete sections — header + body, body + footer, or all three
  • Confirmation dialogs and forms with action rows
  • Post / thread cards with author header and reply footer
  • Settings panels with title + content blocks
  • Any surface where header/footer spacing should be enforced automatically
cancel When NOT to Use
  • Pure content surfaces with no sections — use Paper directly for full layout control
  • As a clickable target — wrap in Button or a custom interactive component instead
  • Nested inside another Card — double borders read busy; use Paper with level="subtle" inside
  • For metrics dashboards — use MetricCard, which is purpose-built for that

toggle_on States

Card is a composition primitive — it has no interactive states (no hover, focus, active, or disabled). When Card needs to feel clickable, wrap it in an interactive parent (Button styled as card, link, MenuItem). Don't add state behavior to Card itself.

biotech Anatomy

Three slots, each rendering only when its prop is provided. The default padding is 16px (tokens.spacing.md). When divided=true, 1px dividers appear between header→body and body→footer.

recommend Best Practices

check_circle Do
  • Use Card when the layout has discrete sections (header + body, body + footer, or all three)
  • Pass any ReactNode as the header slot to compose richer layouts (avatar + name + actions)
  • Use divided=true when section roles are genuinely different (e.g. confirm dialogs)
  • Reach for Paper directly when you need full layout control without slot structure
cancel Don't
  • Don't add hover or click behavior — wrap Card in an interactive component if needed
  • Don't nest Card in Card — double borders read busy. Use Paper level="subtle" instead
  • Don't add box-shadow at rest — surfaces are border-only in TRIO
  • Don't use Card for metrics — that's what MetricCard is for
