Chip

Chips help people enter information, make selections, filter content, or trigger actions. They are compact elements that represent an attribute, text, entity, or action.

category Atomic Component verified Production Ready

info Overview

psychology Purpose
Compact tag-like element for status, filters, removable items, or selected values. Used in filter rows, multi-select displays, and metadata strips.
priority_high Visual Hierarchy
Low-to-medium — Chip is a labeled container with optional interactivity. Static chips are visual; clickable/deletable chips carry weight equal to a small button.
accessibility Accessibility
Interactive chips use role="button" with an action-specific label ("Remove California filter"). Static chips can be plain text content.

rule Usage Guidelines

check_circle When to Use
Correct
  • Use chips for filter tags, categories, and selection states
  • Use default color for neutral informational chips
  • Use primary to indicate active/selected state
  • Use semantic colors (error, warning, success) for status feedback
  • Use the cancel icon (iconRight) when the chip can be removed
  • Use the check icon (iconLeft) to confirm a selected state
cancel When NOT to Use
Incorrect
  • Don't use chips as primary navigation elements (use tabs or buttons)
  • Don't use primary blue for general labels — reserve for selected state
  • Don't use more than 3 semantic color types in one view
  • Don't use chips as standalone call-to-action buttons
  • Don't truncate chip labels — chips should be concise by design

recommend Best Practices

check_circle Do
Correct
  • Semantic colors are for status and feedback only — success, warning, error, info communicate state, not decoration. Active / Pending / Expired chips are correct uses; coloring "RN" or "Float Pool" by team is not.
  • Border radius is 999px (pill) — the rounded shape distinguishes chips from buttons and tags at a glance.
  • When a chip lives inside an Alert, it shares the alert's semantic color — the alert reads as a single semantic block.
  • Use consistent chip sizes within the same context — mixed sizes read as inconsistent system state, not intentional emphasis
  • Keep labels concise (1-2 words ideal) — chips are at-a-glance status indicators
  • Use the cancel icon only when the chip is genuinely removable — otherwise the X promises an interaction that doesn't exist
  • Group related chips together with consistent spacing
cancel Don't
Incorrect
  • Don't use primary blue (#2196F3) for general chip labels — primary blue is reserved for Save/Update actions and links/clickable text. Use color="default" for neutral chips.
  • Don't use semantic colors decoratively — success on a generic tag, warning on a brand label, error as accent — all break the rule that semantic color signals state.
  • Don't recolor chips inside an Alert — the chip inherits the alert's severity color so the block reads as one semantic unit.
  • Don't use chips as call-to-action buttons — chips are status, buttons are actions
  • Don't mix chip sizes in the same row
  • Don't use long labels — keep them concise
  • Don't use more than 3 semantic color types in one view — signal-to-noise collapses
