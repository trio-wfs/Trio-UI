Card

Composed surface built on Paper. Adds header / body / footer slots with consistent section spacing.

view_agenda Overview

Use Card when a layout needs predictable section structure. Use Paper directly when you want full layout control. Card wraps Paper internally — it inherits all Paper props (level, padding, bordered).

No Figma variant axes. The "states" of a Card (with header, with footer, divided) are content presence — handled by React props, not variant explosion.

view_module Anatomy & Usage

Three optional slots. Header and footer render only when provided. Body is always present.

lightbulb Best Practices

  • Use Card when the layout has discrete sections. Header + body, body + footer, or all three. For pure content surfaces without sections, use Paper directly.
  • Reserve divided=true for emphasis. Most cards don't need dividers — they add visual weight. Use dividers when the sections have genuinely different roles (e.g. a confirmation dialog with action buttons).
  • The header slot accepts any ReactNode. Pair it with a flex layout to build composed headers (avatar + name + meta + action) — Card just guarantees the consistent outer spacing.
  • Don't nest Card in Card unless intentional. Double borders read busy. If you want sub-sections, use Paper with level="subtle" inside.
  • Card has no built-in elevation. Follow the Paper rule: no shadows at rest. If a Card needs to feel clickable, wrap it in an interactive component (Button styled as card, MenuItem).
